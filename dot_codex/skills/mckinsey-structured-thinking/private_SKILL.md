# 麦肯锡化思维

用这个 skill 把零散信息整理成清晰、可汇报、可决策的结构。核心不是套模板，而是让表达满足四条原则：结论先行、以上统下、归类分组、逻辑递进。

## 四条原则

1. **结论先行**
   - 先给核心结论或建议，再展开理由、证据和执行动作。
   - 如果事实不足，先给“当前判断 + 待验证点”，不要假装确定。

2. **以上统下**
   - 上层观点必须能概括下层内容。
   - 下层内容必须支撑上层观点，不能游离、重复或只是在堆信息。

3. **归类分组**
   - 同类内容放在同一组。
   - 分组尽量满足 MECE：相互独立、完全穷尽。
   - 如果不能完全 MECE，要说明边界和未覆盖项。

4. **逻辑递进**
   - 组内内容要有顺序，常用顺序包括：重要性、时间、流程、因果、结构、优先级。
   - 不要把不同排序标准混在同一层。

## 工作流程

1. **明确目标**
   - 先判断输出用途：汇报、PRD、方案、复盘、决策建议、问题分析、需求说明。
   - 明确受众：老板、业务方、研发、测试、产品、客户或跨部门评审。

2. **先写一句结论**
   - 用一句话回答“所以我们要做什么/判断是什么/建议是什么”。
   - 如果用户只给了材料，没有明确问题，先提炼最可能的主结论。

3. **拆成 3-5 个一级要点**
   - 每个一级要点回答结论的一个关键支撑面。
   - 一级要点之间避免交叉，尽量互斥。

4. **为每个要点补证据和动作**
   - 每个要点下面放事实、原因、影响、方案、验收或风险。
   - 缺证据时标注“待确认”，不要硬编。

5. **检查结构**
   - 结论是否在最前。
   - 下层是否支撑上层。
   - 同类是否归组。
   - 顺序是否清楚。
   - 是否有遗漏、重复、交叉。

## 输出模板

用于业务方案、PRD、汇报材料时，优先使用：

```text
结论：
一句话说明核心判断或建议。

一、背景/问题
说明为什么现在要处理。

二、目标
说明要达成什么结果。

三、方案
按 3-5 个模块展开，每个模块包含范围、规则、交互、数据或职责。

四、风险与待确认
说明不确定项、依赖项、边界条件。

五、下一步
说明谁在什么时候做什么。
```

用于问题分析时，优先使用：

```text
结论：
当前最主要的问题是……

原因分解：
1. 人/流程/系统/数据/外部因素
2. 每类原因下给事实依据

影响：
说明对效率、成本、质量、风险、体验的影响。

建议：
按优先级给出短期、中期、长期动作。
```

## 使用注意

- 不为了“整齐”牺牲事实准确性。
- 不把 MECE 当教条；业务边界复杂时，先说明分类口径。
- 不堆术语，输出要让目标受众能直接理解和行动。
- 对 PRD 类任务，要把结构落到页面、字段、规则、状态、权限、验收标准上。
